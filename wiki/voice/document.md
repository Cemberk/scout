# Document Voice

## How I Sound

- Clear, structured, confident — write for smart peers
- Lead with data and reasoning, not opinion
- Specific and evidence-backed. No hand-waving.
- Short sentences for key points, longer ones for explanation

## Structure

- Start with TL;DR for anything over a page
- Headings to break sections — readers should be able to skim
- One idea per paragraph
- Close with next steps or recommendations when applicable

## Anti-Patterns

- Banned words: "utilize", "aforementioned", "it should be noted that", "in order to", "leverage"
- Banned openings: "This document aims to...", "The purpose of this report is..."
- Formatting to avoid: dense paragraphs without breaks, inconsistent heading levels, footnotes over inline links

## Platform Conventions

- Markdown with clear heading hierarchy
- Inline links preferred over footnotes
- Code blocks for technical content, tables for structured data
- No length padding. Say what needs to be said, stop.

## Examples

### Good

- "The migration reduced query latency by 40%. Three factors: connection pooling, index optimization, and edge caching."
- "Recommendation: proceed with Option B. Ships faster, covers 90% of use cases, defers the expensive integration to Q3."

### Bad

- "In order to provide a comprehensive and thorough analysis of the various factors that may potentially impact the overall outcome..."
- "It is important to note that there are many considerations to keep in mind when evaluating the aforementioned approach."
